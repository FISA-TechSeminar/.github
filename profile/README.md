# WeMakeIT – 금융권 트래픽 관리 실증 실험 프로젝트

**“폭증하는 금융권 트래픽, 기술로 해결하자!”**  

---

## 📢 프로젝트 소개

- **금융 서비스**는 안정성, 신뢰성이 필수입니다.
- 전통적인 **3-Tier 구조 시스템**은 트래픽 폭증 시 DB, API 서버에 병목과 장애가 쉽게 발생합니다.
- **우리는 개발자 관점에서 실질적인 성능 개선 방법**을 실증적으로 검증했습니다.

## 🔍 실험 목표

1. **실제 금융권과 유사한 3-Tier 구조** 환경에서
2. **캐시 미적용/캐시 Update/캐시 Evict** 등 다양한 전략 적용
3. **k6 + Grafana + Prometheus**로 대용량 트래픽 시뮬레이션
4. **DB 부하, 응답속도, 장애 발생 등 정량적 효과 분석**

---

## 🛠️ 기술 스택

- **Spring Boot** – API 서버 및 비즈니스 로직
- **Redis** – 캐싱/분산락 전략 실험
- **MySQL / Oracle** – 관계형 DB
- **k6** – 대용량 트래픽 부하 테스트
- **Prometheus, Grafana** – 실시간 지표 모니터링/시각화

---

## 📝 주요 실험 시나리오

| 시나리오       | 설명                      | 기대 효과                    |
| -------------- | ------------------------ | ---------------------------- |
| No Cache       | 매 요청 DB 직접 접근     | 병목/지연/장애 위험          |
| Cache Update   | 값 변경 시 캐시 갱신     | 빠른 응답, DB 부하 감소      |
| Cache Evict    | 값 변경 시 캐시 삭제     | 일관성↑, hit/miss 성능차 분석|

> **API**  
> - 계좌 이자 조회, 송금, 상세조회 등 실제 금융 서비스 패턴 기반

---

## 📈 측정 핵심 지표

- **처리량:** 초당 요청 수(RPS), 총 처리 요청 수
- **응답속도:** 평균, 최대, p95 등 분포
- **에러율:** 실패 요청 비율 (성공률)
- **시스템 리소스:** CPU, 메모리 등 (Prometheus 활용)

---

## 🏁 실험 결과 요약

- **캐시 미적용:** DB 과부하, 응답 지연, 장애 발생 가능성 높음
- **캐시 Update/Evict:** 모두 성능 개선 효과
  - 테스트 환경에서는 약 2.5% 성능 차이  
    (트래픽·캐시 미스 폭증 시 차이 더 커질 수 있음)
- **100% 성공률, 평균 39 RPS, 최대 53.5 RPS**  
  → 실제 금융 서비스 요구치에 근접하는 안정성 확인

## 💡 결론 및 인사이트

- 트래픽 관리와 캐싱 설계는 실무에서 ‘돈’이 되는 개발 역량
- 단순 기능구현이 아닌 **시스템 구조·운영 관점**에서의 경험이 중요
- 실험, 분석, 협업 과정을 통해 팀 모두가 성장  
  → **함께 고민하고 직접 검증한 경험**이 가장 값짐
